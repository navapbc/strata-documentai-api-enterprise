# Infrastructure Validators

Validates AWS infrastructure against a committed manifest - no Terraform required on the machine running the check.

## Why this exists

This project's infrastructure is defined in Terraform but **provisioned by clients** who may create resources manually (Console, CLI, scripts) rather than running `terraform apply`.

This validator answers: does the infrastructure match the intended design? It catches configuration mismatches (wrong hash key, missing GSI, incorrect memory size) between what Terraform expects and what actually exists in AWS.

Though originally built for manually-provisioned environments, it's useful anywhere you want to verify live AWS resources without running Terraform.

## Prerequisites

- Python >= 3.11
- [uv](https://docs.astral.sh/uv/) (package manager)
- AWS credentials configured (`~/.aws/credentials` or environment variables)
- Resources tagged with `project`, `stage`, and `component` tags (hard requirement - untagged resources cannot be discovered or validated)

## Architecture

```
tfplan.json (gitignored)        manifest.json (committed)
     │                                │
     │  make infra-manifest           │  make infra-check
     │  (requires TF)                 │  (no TF needed)
     ▼                                ▼
┌──────────┐                   ┌─────────────┐
│ generate │ ───extracts───▶   │  validators │ ───tags API───▶ AWS
└──────────┘  minimal attrs    └─────────────┘    + direct API
```

- **tfplan.json** - Full `terraform show -json` output. Generated by `make infra-expected`. Gitignored.
- **manifest.json** - Minimal, name-free specification grouped by `component` tag. Contains only what validators check (hash_key, gsi_names, memory_size, timeout, count, etc.). Committed.
- **validators/** - Python package. Discovers resources via AWS Resource Groups Tagging API, then validates config against the manifest.

## Key design decisions

- **Name-agnostic**: Clients can name resources however they want. Discovery is purely tag-based (`project` + `stage` + `component`).
- **count field**: When a component has multiple identical resources (e.g. 2 SQS queues), the manifest records `"count": 2` and validators enforce cardinality.
- **UNDISCOVERABLE status**: If tag discovery can't find a component, it's a yellow warning (not a failure). Resources may exist but lack tags.
- **BDA (Bedrock Data Automation) via direct API**: AWSCC-provider resources aren't indexed by the Resource Groups Tagging API. The BDA validator lists projects directly and matches by name pattern.
- **--json clean**: All progress output goes to stderr. Stdout with `--json` is pure JSON for CI piping.

## Workflow

```bash
# One-time setup (via uv)
make infra-setup

# Validate live infra (no TF needed - uses manifest.json)
make infra-check

# Regenerate manifest after TF changes (requires TF)
make infra-manifest

# Lint & format (Terraform + Python)
make infra-lint
make infra-format
```

## Running against another account

The validator works against any account whose resources carry the expected tags. For a different project name or stage value:

```bash
# Different project name
make infra-check PROJECT=myproject

# Stage tag differs from environment (e.g. manual deploy tagged "DEV1")
uv run python -m validators --env dev --stage DEV1 --profile other-account
```

If the target account has never been tagged with `project` + `stage` + `component`, discovery will return nothing and everything reports as missing. The fix is tagging the resources (via `terraform apply` with this branch, or manually) - not a validator issue.

## Untagged resources

The validator reports some "untagged" resources - IAM policies and a Lambda event-source-mapping that carry `project` + `stage` tags but no `component` tag. This is a known TF tagging gap, not a validator gap. These resources are ancillary (access policies generated per-module) and don't affect exit code. Adding `component` tags to them in TF would move them into proper validation.

## Adding a new validator

1. Add the resource type to `generate.py`'s `EXTRACTORS` dict (extract the attributes you'll check).
2. Run `make infra-manifest` to regenerate.
3. Create `validators/your_service.py` with a class inheriting `BaseValidator`.
4. Register the check in `__main__.py`'s `ALL_CHECKS` dict.
5. Add your class to the `InfraValidator` class definition. It uses multiple inheritance - each validator is a mixin, and `InfraValidator` composes them all:

```python
class InfraValidator(
    YourServiceValidator,  # add here
    EcrValidator,
    S3Validator,
    ...
    BaseValidator,  # must be last
):
```
